# TeamPact — Backlog להפיכה למוצר למכירה

**נוצר:** 20.06.2026
**מטרת המסמך:** לרכז את כל מה שצריך כדי להפוך את TeamPact מאפליקציה של מועדון אחד (Team Pact של דודי) למוצר שאפשר למכור למועדוני לחימה אחרים. **לעבוד על זה בשיחה נפרדת**, לא מעורבב עם פיתוח פיצ'רים שוטף.

---

## מצב נוכחי (נקודת הפתיחה)

- **Stack:** React + Vite + Supabase (Postgres + Auth + RLS) + Tailwind. PWA עם Service Worker. Deploy: GitHub `main` → Vercel.
- **Single-tenant:** כל הנתונים שייכים למועדון אחד. אין הפרדה בין מועדונים.
- **תפקידים** נקבעים בשדות על טבלת `profiles`:
  - מתאמן = `role='athlete'`.
  - מאמן = `role='trainer'` + `is_approved=true` (נרשם ב-`/register-coach`, מנהל מאשר ב-`CoachesManager`).
  - מזכירה = מאמן עם `is_secretary=true` + `secretary_branch_id` (מנהל מגדיר ב-`CoachesManager`).
  - מנהל = `is_admin=true` — **אין כפתור בממשק, נקבע ידנית ב-DB (bootstrap)**.

---

## פריט 1 — מסך "ניהול צוות" (ניהול תפקידים מתוך האפליקציה, בלי SQL)

**הבעיה:** היום אי אפשר למנות/להוריד **מנהל** בלי לגעת ישירות ב-DB. גם מינוי מזכירה/אישור מאמן עובד, אבל אין מסך אחד מרוכז וברור.

**מה צריך:**
- מסך "צוות" למנהל: רשימת כל אנשי הצוות (מאמנים, מזכירות, מנהלים) עם אפשרות בלחיצה:
  - לאשר/לדחות מאמן.
  - למנות/להוריד **מזכירה** (+ בחירת סניף).
  - למנות/להוריד **מנהל** (`is_admin`).
  - להשעות/להסיר איש צוות.
- **הגנה:** חייב להישאר תמיד לפחות מנהל אחד (אסור שמנהל יוריד את עצמו ויינעל בחוץ).
- צריך RLS/הרשאות שמתירות רק למנהל לשנות תפקידים.

**עדיין נשאר bootstrap אחד:** המנהל הראשון של כל לקוח חדש — איך נוצר בלי SQL? (ראה פריט 3).

## פריט 2 — Multi-tenant (הפרדת מועדונים)

**הבעיה:** היום DB אחד = מועדון אחד. כדי למכור לכמה לקוחות צריך הפרדה.

**שתי גישות עיקריות, להחליט:**

### גישה א' — DB/פרויקט Supabase נפרד לכל לקוח
- כל מועדון = פרויקט Supabase משלו + deploy משלו (או אותו קוד עם env שונה).
- **יתרון:** הפרדת נתונים מוחלטת, אפס סיכון דליפה בין לקוחות, פשוט יותר לאבטחה.
- **חיסרון:** ניהול תפעולי של הרבה פרויקטים, עלות פר-פרויקט, עדכוני סכמה צריך להריץ על כל אחד.

### גישה ב' — Tenant אחד משותף עם `tenant_id`
- עמודת `tenant_id` (מזהה מועדון) בכל הטבלאות + RLS שמסנן לפי המועדון של המשתמש.
- **יתרון:** תחזוקה אחת, עדכון סכמה פעם אחת, עלות נמוכה.
- **חיסרון:** עבודת רפקטור גדולה (כל טבלה + כל policy + כל שאילתה צריכים tenant_id), וסיכון דליפה אם policy אחד שגוי.

**המלצה ראשונית:** לכמות לקוחות קטנה — גישה א' (פרויקט פר-לקוח) פשוטה ובטוחה יותר להתחיל. למעבר לסקייל — גישה ב'.

**נושאים נלווים ל-multi-tenant:**
- מיתוג פר-מועדון (לוגו, שם, צבעים) — היום הלוגו והשם קבועים (TeamPact).
- סניפים, סוגי מנוי, תחומי לחימה — כבר פר-מועדון בנתונים, אבל צריך לוודא בידוד.

---

## פריט 3 — Onboarding ללקוח חדש (הקמת מנהל ראשון בלי SQL)

**הבעיה:** היום המנהל הראשון נזרע ידנית ב-DB. ללקוח חיצוני זה לא אפשרי.

**מה צריך:**
- תהליך הקמה: לקוח חדש נרשם → נוצר המועדון שלו → **המשתמש הראשון הופך אוטומטית למנהל** של אותו מועדון.
- אפשרויות מימוש: סקריפט/אשף הקמה, או דף "צור מועדון" שמייצר tenant + מגדיר את היוצר כ-`is_admin`.
- תלוי בהחלטת ה-multi-tenant (פריט 2).

---

## פריטים נוספים אפשריים (לדיון בשיחה החדשה)

- **חיוב/מנויים פר-לקוח** — איך גובים מהמועדונים (חודשי/שנתי), ניהול תוכניות.
- **White-label / מיתוג** — דומיין משלו, לוגו, צבעים לכל מועדון.
- **דשבורד-על (super-admin)** לדודi — לראות/לנהל את כל המועדונים-לקוחות.
- **ייבוא נתונים** — איך לקוח חדש מעלה את רשימת המתאמנים הקיימת שלו.
- **תמיכה ו-SLA** — מה קורה כשלקוח צריך עזרה.
