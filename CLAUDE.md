## פרוטוקול תיקון קוד → פרודקשן (חובה!)

כל תיקון קוד חייב להסתיים ברצף סגור. **אם אחד מהשלבים לא בוצע — אסור לסיים את התשובה.**

### השלבים (סדר חובה):

1. **תיקון הקוד בקבצים הרלוונטיים.**
2. **הרצת dev מקומי** — `npm run dev` (ב-`http://localhost:5173`). להגיד למשתמש מפורשות: "רוץ `npm run dev` ובדוק לוקאלית לפני שאני דוחף".
3. **בדיקה לוקאלית של המשתמש** — להמתין לאישור מפורש ("עובד / אישור / דחוף") לפני שממשיכים.
4. **Build מקומי מוצלח** — `npm run build` או `npx vite build` ללא שגיאות.
5. **Commit + Push** — `git add` → `git commit -m "..."` → `git push origin main`.
6. **אימות הגעה לפרודקשן** — להריץ `git log --oneline -3` אחרי ה-push ולוודא שהקומיט החדש שם, ולהזכיר למשתמש לבדוק שה-build ב-Vercel הצליח.
7. **הוראות לטעינה מחדש** — להזכיר למשתמש hard-refresh (Cmd+Shift+R), ואם זה PWA: DevTools → Application → Service Workers → Unregister.

### מה אסור לעשות:

- ❌ אסור לדחוף ל-`main` לפני שהמשתמש אישר שהוא בדק לוקאלית (אלא אם הוא ביקש מפורשות "דחוף ישר").
- ❌ אסור לכתוב "הרץ את הפקודות האלה" בלי לאמת בסוף שהן באמת רצו.
- ❌ אסור להניח שהמשתמש הריץ commit/push בלי לבדוק `git status` או `git log`.
- ❌ אסור לסיים תיקון רב-קבצי בלי TodoList שמסמן כל קובץ + כל שלב.

### מה כן לעשות:

- ✅ אחרי כל תיקון — להציע מפורשות: "רוץ `npm run dev`, בדוק על `localhost:5173`, ותגיד לי אם לדחוף".
- ✅ אם המשתמש לא נתן אישור מפורש לדחיפה, **לחכות**. לא לדחוף לבד.
- ✅ לפני סיום: `git status` + `git log -1` כדי לוודא שאין שינויים תלויים באוויר.
- ✅ אם זה תיקון UI שעובר דרך Service Worker — להזהיר במפורש על cache.

### Bypass (רק כשהמשתמש מבקש מפורשות):

אם המשתמש כתב "דחוף ישר" / "skip local" / "תיקון דחוף לפרודקשן" — אפשר לדלג על שלבים 2-3, אבל **חובה** להזכיר לו שהוא דילג על הבדיקה הלוקאלית.

## בקשות רב-קבציות / רב-ממשקיות

כשהבקשה מזכירה "כל הממשקים", "שלושת הממשקים" (מתאמן/מאמן/מנהל), או "בכל המקומות":
- **ליצור TodoList עם משימה נפרדת לכל ממשק/קובץ** — לא לעבד הכל כמשימה אחת.
- **לסיים בטבלה מסכמת**: קובץ + שורה + מה שונה, לכל הקבצים שנגעו.
- **אם הצלחתי בחלק ולא בכולם — לאמר את זה במפורש** ולא להמשיך הלאה.

## פקודות טרמינל — חובה לציין איפה!

כל פעם שאני נותן פקודת טרמינל, **חובה לציין במפורש באיזו תיקייה להריץ אותה**.

❌ **אסור:** "תריץ `npm run dev`"
✅ **חובה:** "תריץ ב-`/Users/dudibenzaken/teampact-app`:"
```bash
npm run dev
```

או שורה אחת מחוברת:
```bash
cd /Users/dudibenzaken/teampact-app && npm run dev
```

זה תקף לכל פקודה: npm, git, node, python, וכו'. גם אם זו אותה תיקייה כמו פקודה קודמת — לציין שוב.

## SQL / מיגרציות — חובה!

**כל פעם שצריך להריץ SQL ב-Supabase SQL Editor** — לכתוב את כל ה-SQL **ישירות בתוך התשובה בבלוק קוד להעתקה** (```sql ... ```), עם הוראה ברורה "להעתיק ולהדביק ב-SQL Editor".

❌ **אסור לחלוטין** לשלוח רק את הנתיב לקובץ (`/Users/.../migration.sql`) ולצפות מהמשתמש להעתיק ממנו — Supabase SQL Editor מקבל SQL בלבד, לא נתיבים.

✅ גם אם הקובץ נשמר בריפו (לתיעוד), עדיין **חובה** להציג את התוכן המלא בתשובה כקופי-פייסט מוכן.
