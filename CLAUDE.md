## כלל ברזל: אבחון = ראיות מהשרת, לא ניחוש מהקוד (חובה!)

**נלמד ביוקר ב-15.07.2026 (בעיית ה-egress): שלוש פעמים ברצף אבחנתי "אשם" לפי קריאת קוד בלבד ("זה נראה כאילו זה מושך יותר מדי"), תיקנתי, נדחף — והמספר האמיתי בדאשבורד בכלל לא זז.** קריאת קוד היא ניחוש, לא בדיקה. היא טובה כדי *לחפש חשודים*, אבל אסור לקבוע אבחנה סופית או להציע תיקון על סמך זה בלבד.

**הכלל:** לפני שמציעים תיקון לכל בעיה שקשורה לביצועים/עומס/צריכה (egress, זמן טעינה, עומס DB, שגיאות בפרודקשן) — **חובה לאמת מול ראיה חיה מהשרת** לפני שכותבים שורת קוד אחת:
- Supabase: Logs Explorer (edge_logs/postgres_logs) עם שאילתת SQL אמיתית שסופרת בקשות בפועל, לא הנחה.
- דאשבורד Usage — לקרוא מספרים אמיתיים (hover על הגרף), לא להעריך.
- אם יש ספק לגבי מקור תקלה בפרודקשן — Chrome MCP לתעד בפועל מה קורה במסך, לא רק לקרוא קוד.

**אחרי כל תיקון — חובה לחזור ולמדוד מול השרת (לא להניח שהתיקון עבד כי ה-build עבר או כי "זה הגיוני").** אם המספר לא זז — האבחנה הייתה שגויה, לחזור לשלב הראיות ולא לעבור לתיקון הבא בניחוש.

## סביבות: staging → production (חובה!)

**כלל ברזל: כל שינוי נבדק קודם בסביבת בדיקה, ורק אחרי שעובד — נדחף לפרודקשן.** אסור לדחוף ישר ל-`main` בלי בדיקה, אלא אם דודי ביקש מפורשות "דחוף ישר".

הסדר הנכון:
1. **תיקון קוד** בקבצים.
2. **בדיקה בסביבת בדיקה** — לוקאלי (`npm run dev` על `localhost:5173`) ו/או branch `staging`.
3. **רק אחרי אישור "עובד"** — מיזוג/דחיפה ל-`main` (Vercel מעלה אוטומטית לפרודקשן).

מצב הסביבות (נכון ל-15.07.2026, מאומת ישירות מול package.json בפועל — לא להניח, לבדוק `cat package.json` אם יש ספק, כי זה השתנה בעבר בלי שהתיעוד עודכן):
- **לוקאלי** — עובד. `npm run dev` (בלי מוד) → `.env.local` → **DB פרודקשן** (`pnicoluujpidguvniwub`), בלי באנר. **אין סקריפט `dev:prod`** — אם מישהו מציע את זה, זו טעות מתיעוד ישן. לבדיקה מול staging: `npm run dev:staging`.
- **staging** (branch `staging`, `VITE_APP_ENV=staging`) — קיים בקוד אבל **לא מוכן**: חסרים ב-Vercel `VITE_SUPABASE_URL`/`VITE_SUPABASE_ANON_KEY` ל-staging + DB נפרד, ולכן הכתובת `teampact-app-git-staging-…vercel.app` מחזירה דף לבן (`src/lib/supabase.js` זורק שגיאה מכוונת בלי env, כדי לא לגעת ב-DB הפרודקשן). **להשלים לפני שימוש בו כסביבת בדיקה.**
- **production** — branch `main` → Vercel auto-deploy → teampact-app.vercel.app. DB: `pnicoluujpidguvniwub.supabase.co`.

לפני דחיפה: לוודא על איזה branch אנחנו (`git branch --show-current`), ולשאול את דודי אם לא ברור אם היעד `staging` או `main`.

## פרוטוקול תיקון קוד → פרודקשן (חובה!)

כל תיקון קוד חייב להסתיים ברצף סגור. **אם אחד מהשלבים לא בוצע — אסור לסיים את התשובה.**

### השלבים (סדר חובה):

1. **תיקון הקוד בקבצים הרלוונטיים.**
2. **הרצת dev מקומי** — `npm run dev` (ב-`http://localhost:5173`). להגיד למשתמש מפורשות: "רוץ `npm run dev` ובדוק לוקאלית לפני שאני דוחף".
3. **בדיקה לוקאלית של המשתמש** — להמתין לאישור מפורש ("עובד / אישור / דחוף") לפני שממשיכים.
4. **Build מקומי מוצלח** — `npm run build` או `npx vite build` ללא שגיאות.
5. **Commit + Push** — `git add` → `git commit -m "..."` → `git push origin main`.
6. **אימות הגעה לפרודקשן** — להריץ `git log --oneline -3` אחרי ה-push ולוודא שהקומיט החדש שם, ולהזכיר למשתמש לבדוק שה-build ב-Vercel הצליח.
7. **הוראות לטעינה מחדש** — להזכיר למשתמש hard-refresh (Cmd+Shift+R), ואם זה PWA: DevTools → Application → Service Workers → Unregister.

### מה אסור לעשות:

- ❌ אסור לדחוף ל-`main` לפני שהמשתמש אישר שהוא בדק לוקאלית (אלא אם הוא ביקש מפורשות "דחוף ישר").
- ❌ אסור לכתוב "הרץ את הפקודות האלה" בלי לאמת בסוף שהן באמת רצו.
- ❌ אסור להניח שהמשתמש הריץ commit/push בלי לבדוק `git status` או `git log`.
- ❌ אסור לסיים תיקון רב-קבצי בלי TodoList שמסמן כל קובץ + כל שלב.

### מה כן לעשות:

- ✅ אחרי כל תיקון — להציע מפורשות: "רוץ `npm run dev`, בדוק על `localhost:5173`, ותגיד לי אם לדחוף".
- ✅ אם המשתמש לא נתן אישור מפורש לדחיפה, **לחכות**. לא לדחוף לבד.
- ✅ לפני סיום: `git status` + `git log -1` כדי לוודא שאין שינויים תלויים באוויר.
- ✅ אם זה תיקון UI שעובר דרך Service Worker — להזהיר במפורש על cache.

### Bypass (רק כשהמשתמש מבקש מפורשות):

אם המשתמש כתב "דחוף ישר" / "skip local" / "תיקון דחוף לפרודקשן" — אפשר לדלג על שלבים 2-3, אבל **חובה** להזכיר לו שהוא דילג על הבדיקה הלוקאלית.

**בכל שינוי קוד — לבדוק ולוודא תקינות בכל ארבעת הממשקים, גם אם הבקשה הזכירה רק אחד:**

לפני סיום: לעבור ממשק-ממשק ולשאול "האם השינוי הזה משפיע כאן? אם כן — האם נבדק?". אם ממשק לא רלוונטי — לציין זאת מפורשות ("מזכירה: לא רלוונטי כי הטאב מוסתר"), לא לדלג בשתיקה.

כשהבקשה מזכירה "כל הממשקים", "ארבעת/שלושת הממשקים", או "בכל המקומות":
- **ליצור TodoList עם משימה נפרדת לכל ממשק/קובץ** — לא לעבד הכל כמשימה אחת.
- **לסיים בטבלה מסכמת**: ממשק + קובץ + שורה + מה שונה / "לא רלוונטי", לכל 4 הממשקים.
- **אם הצלחתי בחלק ולא בכולם — לאמר את זה במפורש** ולא להמשיך הלאה.

## פקודות טרמינל — חובה לציין איפה! (כלל קשיח)

כל פקודת טרמינל שאני נותן למשתמש **חייבת להופיע כשורה אחת מחוברת עם `cd` בתוך בלוק הקוד עצמו**. הנתיב חייב להיות *בתוך* הבלוק שמעתיקים — לא כטקסט נפרד לפני הבלוק.

❌ **אסור — נתיב כטקסט והפקודה לבד בבלוק:**
> "תריץ ב-`/Users/dudibenzaken/teampact-app`:"
> ```bash
> npm run dev
> ```
(זו בדיוק התקלה החוזרת — אי אפשר להעתיק בשורה אחת.)

✅ **חובה — `cd` + הפקודה באותו בלוק, מוכן להעתקה:**
```bash
cd /Users/dudibenzaken/teampact-app && npm run dev
```

כללי ברזל:
- כל בלוק `bash` שמיועד למשתמש פותח ב-`cd /Users/dudibenzaken/teampact-app &&`.
- תקף לכל פקודה: npm, git, node, npx, python וכו'. גם אם זו אותה תיקייה כמו קודם — לחזור על ה-`cd` בכל בלוק.
- אם יש כמה פקודות רצופות — או לשרשר ב-`&&`, או שכל בלוק נפרד מתחיל ב-`cd`.
- לפני שליחת תשובה עם פקודות טרמינל: לסרוק שכל בלוק `bash` מתחיל ב-`cd <נתיב>`. אם לא — לתקן לפני השליחה.

**כל פעם שצריך להריץ SQL ב-Supabase SQL Editor** — לכתוב את כל ה-SQL **ישירות בתוך התשובה בבלוק קוד להעתקה** (```sql ... ```), עם הוראה ברורה "להעתיק ולהדביק ב-SQL Editor".

❌ **אסור לחלוטין** לשלוח רק את הנתיב לקובץ (`/Users/.../migration.sql`) ולצפות מהמשתמש להעתיק ממנו — Supabase SQL Editor מקבל SQL בלבד, לא נתיבים.

✅ גם אם הקובץ נשמר בריפו (לתיעוד), עדיין **חובה** להציג את התוכן המלא בתשובה כקופי-פייסט מוכן.
